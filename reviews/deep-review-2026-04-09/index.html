    Deep Architecture Review: icom-lan¶

    Date: 2026-04-09 Reviewer: Claude Opus 4.6 (automated deep review) Commit: 9bc2d1b (main) Source: 99 Python files in src/icom_lan/ (45,034 LOC) Tests: 155 test files

    Executive Summary¶

    icom-lan is a well-structured, ambitious project that reverse-engineers a proprietary UDP protocol and exposes it through a clean multi-backend architecture. The protocol layer is solid, the decomposition into capability protocols is well-designed, and the test coverage is generally good.

    However, there are significant issues in three areas: (1) god modules that have grown past maintainability thresholds, (2) concurrency patterns with subtle race conditions and fire-and-forget risks, and (3) defensive exception handling that silently swallows failures in critical paths. These are typical for a fast-moving project of this complexity and are all fixable.

    Overall verdict: needs_attention

    Findings¶

    [Severity: P1] commands.py is a 6830-line god module with 312 functions¶

    File: src/icom_lan/commands.py Problem: A single flat module containing every CI-V command builder and parser (312 functions, 6830 lines). No internal organization beyond ordering. Every new radio command adds two more functions to this file. Impact: Extremely difficult to navigate, review, or test in isolation. Import times grow linearly. Changes to one command family risk merge conflicts with unrelated work. Fix: Split into commands/ package by command family: commands/frequency.py, commands/mode.py, commands/scope.py, commands/levels.py, commands/system.py, commands/memory.py. Re-export from commands/__init__.py for backwards compatibility.

    [Severity: P1] radio.py CoreRadio is a 4593-line class with 334 methods¶

    File: src/icom_lan/radio.py Problem: CoreRadio has 334 methods, 60+ instance attributes initialized in __init__, and directly implements every capability protocol. Despite decomposition of control-phase, civ-rx, and audio-recovery into runtimes, the class itself remains a monolith that mixes lifecycle management, command dispatch, audio control, scope control, dual-receiver routing, VFO management, and state caching. Impact: Any change to CoreRadio risks breaking unrelated functionality. The massive import section (lines 1-300) is itself a code smell. Testing requires mocking dozens of internal attributes. Fix: Continue the runtime decomposition pattern. Extract scope methods into _scope_runtime.py, audio methods into _audio_runtime.py (beyond just recovery), and dual-receiver routing into _dual_rx_runtime.py. CoreRadio becomes a thin composition root.

    [Severity: P1] TOCTOU race in UDP port allocation¶

    File: src/icom_lan/_control_phase.py:143-150 Problem: The code binds a temporary UDP socket to get a free port, then immediately closes it and hopes the port is still free when the real CI-V/audio transport binds later: 

    _civ_sock = _socket.socket(_socket.AF_INET, _socket.SOCK_DGRAM)
_civ_sock.bind((local_bind_host, 0))
_civ_local_port = _civ_sock.getsockname()[1]
_civ_sock.close()  # Port is now free for anyone to grab
    Between close() and the actual transport connect() (line 218), another process could steal the port. Impact: Intermittent OSError: Address already in use during connect, especially under high system load or when multiple instances connect simultaneously. Fix: Pass the already-bound socket fd directly to the transport via socket.fromfd() or SO_REUSEADDR, or use local_port=0 and let the transport bind once.

    [Severity: P1] AudioSubscription.stop() fires untracked task¶

    File: src/icom_lan/audio_bus.py:106 Problem: stop() is synchronous but needs to call async _remove_subscriber(). It does: 

    asyncio.create_task(self._bus._remove_subscriber(self))
    This task is never stored, never awaited, and never checked. If the event loop is closing or the task raises, it's silently lost. Impact: Subscriber leak (never removed from bus), audio RX never stops (last subscriber thinks it's gone but bus still has a reference), potential RuntimeWarning: coroutine was never awaited if no event loop is running. Fix: Store the task and await it in __aexit__, or make stop() async. Since __aexit__ already exists and is async, it should call await self._bus._remove_subscriber(self) directly instead of delegating to stop().

    [Severity: P1] 61 bare assert resp is not None in radio.py¶

    File: src/icom_lan/radio.py (61 occurrences) Problem: After _send_civ_raw() returns, the code asserts the response is not None: 

    resp = await self._send_civ_raw(civ)
assert resp is not None
freq = parse_frequency_response(resp)
    If Python is run with -O (optimizations), all asserts are stripped. Even without -O, AssertionError is not a CommandError or TimeoutError, so callers can't catch it meaningfully. Impact: In production with PYTHONOPTIMIZE=1, resp being None passes silently to the parser, causing cryptic TypeError or AttributeError instead of a clear error. The 61 identical patterns also suggest a missing abstraction. Fix: Replace with explicit checks: 
    resp = await self._send_civ_raw(civ)
if resp is None:
    raise CommandError("No response for get_freq")
    Or better, create _send_civ_expect() that wraps this pattern once.

    [Severity: P2] _civ_rx.py has 12 broad except Exception blocks¶

    File: src/icom_lan/_civ_rx.py (lines 311, 333, 352, 362, 475, 672, 1013, 1023, 1041, 1096, 1104, 1237) Problem: The CI-V receive pump catches Exception broadly and logs at DEBUG level in many places: 

    except Exception:
    logger.debug("civ-rx: cache update failed", exc_info=True)
    This means programming errors (KeyError, TypeError, AttributeError from wrong frame parsing) are silently swallowed during normal operation. Only visible with DEBUG logging. Impact: Latent bugs in CI-V parsing are invisible. A malformed frame that triggers a real bug will be logged at DEBUG and never noticed. The CI-V stream appears healthy while state updates silently fail. Fix: Catch specific expected exceptions (ValueError, struct.error, IndexError for malformed frames). Log unexpected exceptions at WARNING or ERROR level. Consider a counter metric for parse failures.

    [Severity: P2] __init__.py exports 150+ symbols including low-level internals¶

    File: src/icom_lan/__init__.py Problem: The top-level package exports 150+ symbols including build_civ_frame, parse_header, serialize_header, bcd_encode, bcd_decode, CONTROLLER_ADDR, and 40+ scope sub-command functions. These are implementation details, not public API. Impact: Massively inflated public API surface. Any change to internal wire format functions becomes a breaking change. Import from icom_lan import * pulls in an overwhelming number of symbols. Fix: Reduce __all__ to the true public API: create_radio, Radio, capability protocols, exceptions, types/enums, and RadioProfile. Move internal re-exports to a icom_lan.compat or icom_lan.internals module for users who need low-level access.

    [Severity: P2] _control_phase.py hardcodes "IC-7610" in conninfo¶

    File: src/icom_lan/_control_phase.py:452 Problem: 

    conninfo = build_conninfo_packet(
    ...
    radio_name="IC-7610",
    ...
)
    The radio name is hardcoded as "IC-7610" regardless of what radio is actually being connected to. Impact: Works for IC-7610 but may cause issues with other Icom radios that validate the radio_name field in the conninfo packet. Misleading in logs and diagnostics. Fix: Use self._host._profile.model or pass the model name through the host protocol.

    [Severity: P2] ControlPhaseHost protocol uses Any for critical fields¶

    File: src/icom_lan/_runtime_protocols.py:114,134,167-170 Problem: Several fields in ControlPhaseHost are typed as Any: 

    _conn_state: Any        # Should be RadioConnectionState
_audio_codec: Any       # Should be AudioCodec
_audio_stream: Any      # Should be AudioStream | None
_pcm_tx_fmt: Any
_pcm_rx_user_callback: Any
_opus_rx_user_callback: Any
    Impact: mypy cannot verify that runtimes use these fields correctly. Type errors in the control phase / audio recovery paths are invisible to static analysis. Fix: Import the proper types (conditionally under TYPE_CHECKING) and use them.

    [Severity: P2] Excessive getattr() defensive patterns in radio.py¶

    File: src/icom_lan/radio.py (6 occurrences), _civ_rx.py (20 occurrences), _control_phase.py (6 occurrences) Problem: Code frequently uses getattr(self, "_attr", None) to access attributes that are always set in __init__: 

    _rs = getattr(host, "_radio_state", None)  # Always exists
last = getattr(self, "_last_civ_data_received", None)  # Always exists
    Impact: Masks bugs — if an attribute is missing it should be a clear AttributeError, not a silent None fallback. Makes the code harder to reason about (reader can't tell if the attribute might genuinely not exist or if it's just defensive coding). Fix: Use direct attribute access for attributes guaranteed by __init__. Reserve getattr() for truly optional attributes (e.g., on objects that might be mocks).

    [Severity: P2] _connected / _intentional_disconnect backwards-compat shims¶

    File: src/icom_lan/radio.py:716-737 Problem: CoreRadio maintains property shims that translate between the old boolean flags and the new RadioConnectionState enum: 

    @property
def _connected(self) -> bool:
    return self._conn_state == RadioConnectionState.CONNECTED

@_connected.setter
def _connected(self, value: bool) -> None:
    if value:
        self._conn_state = RadioConnectionState.CONNECTED
    ...
    Impact: Two parallel representations of the same state. Internal code can set _connected = False and get a different result than setting _conn_state = DISCONNECTED. The setter for _intentional_disconnect has surprising behavior: setting it to False transitions to RECONNECTING, not CONNECTED. Fix: Migrate all internal callers to use _conn_state directly. Remove the property shims. Update tests that rely on the boolean interface.

    [Severity: P2] web/server.py has 17 getattr() calls suggesting loose coupling¶

    File: src/icom_lan/web/server.py Problem: The web server uses getattr() extensively to probe the radio object for capabilities: 

    getattr(self._radio, "_profile", None)
getattr(self._radio, "state_cache", None)
    These suggest the web server doesn't trust the type of self._radio, even though it's typed as Radio. Impact: The web server is effectively untyped — mypy cannot verify any of these accesses. Refactoring the Radio protocol won't produce type errors in the web server. Fix: Use isinstance() checks against capability protocols where appropriate, and direct attribute access for core Radio methods.

    [Severity: P2] _receive_guid() busy-loops with 30 iterations¶

    File: src/icom_lan/_control_phase.py:426-437 Problem: 

    async def _receive_guid(self) -> bytes | None:
    await asyncio.sleep(0.3)
    guid = None
    for _ in range(30):
        try:
            d = await h._ctrl_transport.receive_packet(timeout=0.1)
            if len(d) == CONNINFO_SIZE:
                guid = d[0x20:0x30]
        except asyncio.TimeoutError:
            break
    return guid
    This drains up to 30 packets looking for one with a specific size. Packets that don't match are silently discarded. Impact: During high-traffic scenarios (e.g., radio already streaming scope data), important control packets could be consumed and dropped by this loop. The 0.3s sleep + up to 3s of polling adds latency to connection setup. Fix: Use a filtered receive that matches on packet size, or buffer non-matching packets for later processing.

    [Severity: P2] web/handlers.py is 2399 lines with mixed concerns¶

    File: src/icom_lan/web/handlers.py Problem: Contains ControlHandler (WebSocket command dispatch), ScopeHandler (scope data streaming), AudioBroadcaster (audio fan-out), and AudioHandler (bidirectional audio WebSocket). Each is a substantial class with its own lifecycle, but they're all in one file. Impact: Hard to navigate, test, or modify one handler without risking others. The file will only grow as new WebSocket endpoints are added. Fix: Split into handlers/control.py, handlers/scope.py, handlers/audio.py.

    [Severity: P2] radio_poller.py uses sentinel dataclasses instead of proper commands¶

    File: src/icom_lan/web/radio_poller.py Problem: The command queue uses empty dataclasses as sentinel values: 

    @dataclass
class EnableScope: pass

@dataclass
class DisableScope: pass
    These are put into a generic asyncio.Queue with no type safety. Impact: Minor — it works, but is an ad-hoc command pattern. No compile-time guarantee that the queue consumer handles all command types. Fix: Use a tagged union / enum-based command pattern, or at minimum a Command base class.

    [Severity: P3] AudioBus.__init__ types radio as Any¶

    File: src/icom_lan/audio_bus.py:169 Problem: 

    def __init__(self, radio: Any, *, jitter_depth: int = 5) -> None:
    Impact: No type checking on the radio parameter. Could pass any object. Fix: Type as AudioCapable (with conditional import under TYPE_CHECKING).

    [Severity: P3] sync.py hardcodes radio_addr=0x98 default¶

    File: src/icom_lan/sync.py:56 Problem: The synchronous wrapper hardcodes radio_addr=0x98 (IC-7610) as default, while the async CoreRadio defaults to None and uses profile resolution. Impact: Sync users connecting to non-IC-7610 radios must always specify radio_addr. Inconsistent API between sync and async interfaces. Fix: Match the async signature — default to None and let profile resolution handle it.

    [Severity: P3] ConnectionError and TimeoutError shadow builtins¶

    File: src/icom_lan/exceptions.py:20,33 Problem: Custom exceptions named ConnectionError and TimeoutError shadow Python builtins of the same name. Impact: Inside any module that imports these, except ConnectionError catches only the icom-lan version, not stdlib ConnectionError. This is a known design choice (documented in the codebase), but still a footgun for contributors. Fix: Consider IcomConnectionError / IcomTimeoutError in a future major version, or at minimum document the shadowing prominently in the exceptions module.

    [Severity: P3] IcomRadio subclass is essentially empty¶

    File: src/icom_lan/radio.py:4556-4566 Problem: 

    class IcomRadio(CoreRadio):
    """LAN adapter for IC-7610 built on top of the shared executable core."""

    @staticmethod
    async def _flush_queue(transport, max_pkts=200):
        from ._control_phase import ControlPhaseRuntime
        return await ControlPhaseRuntime._flush_queue(transport, max_pkts)

    pass
    The entire class is a CoreRadio with one static method override and a pass statement. Impact: The CoreRadio / IcomRadio split suggests a future where CoreRadio is backend-neutral, but currently CoreRadio itself is fully LAN-specific (references UDP transports, ports, etc.). The split is premature. Fix: Either merge back into IcomRadio (since CoreRadio is LAN-specific anyway), or actually make CoreRadio backend-neutral by extracting LAN-specific transport setup.

    [Severity: P3] Protocol compliance check uses IcomRadio(host="") instances¶

    File: src/icom_lan/radio.py:4574-4593 Problem: 

    def _check_protocol_compliance() -> None:
    assert isinstance(IcomRadio(host=""), Radio)
    assert isinstance(IcomRadio(host=""), AudioCapable)
    Creates actual radio instances (allocating transports, state caches, etc.) just to check isinstance. Impact: Wasteful; creates real IcomTransport objects. Could fail if constructor has side effects. Not run in tests automatically. Fix: Add this as a proper test in tests/test_radio.py. Use @pytest.mark.parametrize with all protocol types.

    [Severity: P1] radio_poller.py _execute() is 920 lines with 100+ match branches¶

    File: src/icom_lan/web/radio_poller.py:657-1577 Problem: Single massive match statement dispatching all radio commands. Each branch has duplicated receiver-switching logic, optimistic state updates, and event publishing. Impact: Impossible to modify without affecting multiple branches. High copy-paste bug risk. Untestable per-command. Fix: Extract to a command dispatcher table: handlers = {SetFreq: self._handle_set_freq, ...}.

    [Severity: P1] web/handlers.py has 9 nearly-identical _enqueue_rc_* methods¶

    File: src/icom_lan/web/handlers.py:1037-1800 Problem: 9 handler methods (frequency, power, DSP, audio, scope, antenna, system, memory, misc) share identical structure: match on command name, parse params, check capability, build dataclass, queue. 67 similar case/match patterns. Impact: Changes to validation logic must be replicated 9 times. Bugs exist in some methods but not others. Fix: Create a handler registry/factory eliminating structural duplication.

    [Severity: P2] dict[str, Any] dominates web handler types (32 occurrences)¶

    File: src/icom_lan/web/handlers.py Problem: All HTTP/WebSocket message handlers use dict[str, Any] instead of TypedDict or Protocol-based message types. No schema validation at type-check time. Impact: Runtime errors on malformed messages, no IDE autocomplete, refactoring hazard. Fix: Create TypedDict classes for each message type.

    [Severity: P2] Unbounded retransmit buffer in transport¶

    File: src/icom_lan/transport.py (tx_buffer) Problem: self.tx_buffer: OrderedDict[int, bytes] = OrderedDict() has no size limit. If radio doesn't ACK, buffer grows without bound. Impact: Memory leak under poor network conditions. Fix: Add max_tx_buffer_size and drop oldest when exceeded.

    [Severity: P2] Packet queue overflow race condition in transport¶

    File: src/icom_lan/transport.py:580-605 Problem: Between full() check and put_nowait(), another task can drain the queue. The recovery logic doesn't handle this race atomically. Impact: Non-scope packets can be silently dropped during high load. Fix: Use a single atomic operation or wrap both check+put in a lock.

    [Severity: P2] No lock on connection state transitions¶

    File: src/icom_lan/_control_phase.py:280-285 Problem: disconnect() checks _conn_state != CONNECTED and then sets DISCONNECTING without a lock. A concurrent connect/watchdog can observe inconsistent state. Impact: Double-disconnect, leaked tasks, inconsistent state machine. Fix: Use an asyncio.Lock or atomic state transitions.

    [Severity: P2] CI-V event queue overflow silently drops events¶

    File: src/icom_lan/_civ_rx.py:1044-1051 Problem: Events silently dropped when queue full — no logging, no counter, no backpressure signal. Impact: Event consumers (web server, poller) miss critical state changes without notification. Fix: Add a dropped counter and log periodically.

    [Severity: P3] _civ_rx.py hard-codes IC-7610-specific routing tables¶

    File: src/icom_lan/_civ_rx.py:49-90 Problem: _CMD14_RECEIVER_LEVEL_FIELDS, _CMD16_RECEIVER_BOOL_FIELDS etc. are hard-coded dictionaries assuming IC-7610 sub-command layout rather than being data-driven from rig profiles. Impact: Adding support for a new Icom radio requires editing _civ_rx.py instead of just adding a TOML profile. Fix: Move routing tables into RadioProfile and look them up at runtime.

    Test Quality Assessment¶

    Coverage Gaps (Critical)¶

    1. 7 source modules have ZERO test files:
    2. radio_protocol.py (876 lines) — defines the core protocol interface
    3. backends/_icom_serial_base.py (590 lines) — serial backend base
    4. capabilities.py (195 lines) — capability registry
    5. meter_cal.py (117 lines) — meter calibration
    6. audio/lan_stream.py (703 lines) — LAN audio streaming
    7. backends/yaesu_cat/ (all modules) — entire Yaesu backend

    8. command_map.py (48 lines) — command mapping

    9. Transport failure scenarios untested — no tests for socket errors, handshake timeouts, invalid radio responses.

    10. Protocol edge cases missing — BCD boundary conditions (max digits, overflow), packet type identification for all 8+ types, sequence number wraparound at 0xFFFF.

    11. Pre-existing failing tests noted in MEMORY.md: TestBackpressure::test_scope_*, TestControlChannel::test_hello_capabilities_list, TestControlChannel::test_state_snapshot_fields.

    Flaky Patterns¶

    1. Sleep-based timing in reconnect tests — test_reconnect.py uses asyncio.sleep(0.6) to wait for watchdog timeouts. Breaks under system load.

    2. Audio stress test uses real blocking sleep — test_audio_stress.py:109 calls time.sleep() inside callback, runs for 3+ real seconds.

    3. Thread contention tests not deterministic — test_audio_buffer_pool.py:125 uses time.sleep(0.001) hoping for race conditions.

    Anti-patterns¶

    1. Mock-over-behavior — test_audio_recovery.py verifies mock call arguments but not that audio actually flows end-to-end.

    2. conftest.py monkey-patches _handle_raw_civ_packet — masks missing implementation with a no-op lambda.

    3. Integration tests require real hardware — 20 integration test files skip silently without physical radio. No mock equivalents for CI.

    4. MockTransport doesn't validate protocol — accepts any bytes, doesn't parse CI-V frames or simulate radio behavior.

    Test Quality Observations¶

    • Protocol tests (test_protocol.py, test_commands.py) are excellent — testing real byte sequences against known values.
    • Transport tests properly mock UDP and verify handshake sequences.
    • Radio tests tend to mock many internal attributes, making them brittle to refactoring.
    • Performance regression tests have no stored baselines — assertions are loose (e.g., ops_per_sec > 50_000).
