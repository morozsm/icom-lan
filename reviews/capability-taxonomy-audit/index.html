    Capability Taxonomy Audit¶

    Date: 2026-04-01 Issue: #447 — Capability taxonomy audit and unified capabilities.py module Epic: #446 — Unified capability gating system

    Summary¶

    The codebase uses 3 inconsistent patterns for checking radio capabilities:

    1. isinstance(radio, AdvancedControlCapable) — monolithic Protocol class check (80+ sites)
    2. hasattr(radio, "method_name") — duck typing (20+ sites)
    3. "tag" in radio.capabilities — correct tag-based pattern (only 2 sites in production code)

    This audit maps every capability check to the tag that should replace it in Phase 2.

    Part 5: Naming Inconsistencies¶

    Issue Details Recommendation
    dual_rx vs dual_watch Separate tags, correct. dual_rx = two independent receivers; dual_watch = monitoring sub-band. IC-7610 has both; FTX-1 has dual_rx only. Keep both — semantically distinct.
    agc missing from capabilities [agc] section exists in 5 TOML profiles; hasattr(radio, "set_agc") used in radio_poller:1732. No capability tag. Add agc tag in Phase 2.
    tuning_step missing get_tuning_step() gated by AdvancedControlCapable (handlers:937). No capability tag. Add tuning_step tag in Phase 2.
    band_edge missing get_band_edge_freq() gated by AdvancedControlCapable (handlers:951). No capability tag. Add band_edge tag in Phase 2.
    xfc missing get_xfc_status() gated by AdvancedControlCapable (handlers:958). No capability tag. Add xfc tag in Phase 2.
    System methods overloaded on power_control get_system_date, get_system_time, get_utc_offset, get_ref_adjust, get_civ_transceive all gated by AdvancedControlCapable. Could share power_control or need a new system tag. Consider system_settings tag or keep under power_control.

    Part 7: Recommendations for Phase 2 (#433)¶

    1. Replace all isinstance(radio, AdvancedControlCapable) with specific tag checks. Each site maps to a distinct capability tag (see tables above). This is the highest-value change — it turns a single monolithic gate into fine-grained per-feature gating.

    2. Replace hasattr(radio, "method") with tag checks. The hasattr pattern is fragile (typos, renames). Tag-based gating is the canonical pattern.

    3. Replace other Protocol isinstance checks (AudioCapable, ScopeCapable, etc.) with tag checks where the radio has a .capabilities set. Keep runtime_helpers.py hybrid validation as-is (it handles radios that lack profile-based capabilities).

    4. Add new capability tags to capabilities.py and TOML profiles:

    5. agc — AGC mode selection
    6. tuning_step — tuning step control
    7. band_edge — band edge frequency query

    8. xfc — transverter function status

    9. Consider a system_settings tag to group system-level features (date, time, UTC offset, ref adjust, CIV transceive) that are currently lumped under power_control.

    10. Migration order: Start with radio_poller.py (already has self._caps infrastructure), then handlers.py, then cli.py, then sync.py and rigctld/handler.py.
