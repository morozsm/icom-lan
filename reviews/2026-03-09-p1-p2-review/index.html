    🔴 Critical Issues¶

    C1: ~~$state() Used Outside Svelte Component~~ ⚠️ INCORRECT FIX — REVERTED¶

    File: frontend/src/lib/utils/smoothing.svelte.ts
    Line: 6

    let current = $state(0);  // ✅ CORRECT (contrary to original review)

    Original problem (WRONG): Review claimed $state() doesn't work in .ts files.

    Reality: .svelte.ts is a special Svelte 5 suffix that ALLOWS runes! - .svelte.ts files are compiled by Svelte preprocessor - Runes ($state, $derived, $effect) are explicitly supported in .svelte.ts - This is the recommended pattern for shared reactive logic in Svelte 5

    What actually happened: 1. Review incorrectly flagged $state(0) as an error 2. Fix commit d55ae7e changed $state(0) → 0 (plain variable) 3. This broke S-meter reactivity — needle stopped moving 4. Reverted in follow-up commit (2026-03-09 14:27 EDT)

    Correct code: 

    export function createSmoother(attack = 0.12, release = 0.32) {
  let current = $state(0);  // ✅ Reactive state for Svelte 5
  // ...
  return {
    get value() { return current; },  // Getter exposes reactive value
  };
}

    Why it works: - .svelte.ts !== .ts — different treatment by Svelte compiler - .svelte files: runes work in <script> blocks - .svelte.ts files: runes work at top level (for shared modules) - .ts files: runes DON'T work (plain TypeScript)

    Impact of incorrect fix: S-meter frozen (no needle movement) — critical UX bug.

    Lesson learned: Always verify Svelte 5 conventions before "fixing" runes usage. .svelte.ts is a first-class reactive module format.

    C2: Waterfall pushRow Memory Allocation Regression¶

    File: frontend/src/lib/renderers/waterfall-renderer.ts
    Line: 137-138

    const rowBuf = ctx.createImageData(w, 1);  // ❌ Allocates EVERY frame
const rowData = rowBuf.data;

    Problem: Creates new ImageData object every frame (30+ fps) instead of reusing this.rowBuf initialized in _initBuffers().

    Impact: - ~1-2 KB allocation per frame × 30 fps = 30-60 KB/sec garbage - Forces GC pauses, drops frames on mobile

    History: This was fixed on 2026-03-08 (commit 8d5488f) but regressed in 5d3de0e when waterfall agent rewrote the renderer.

    Fix: 

    if (!this.rowBuf || this.rowBuf.width !== w) {
  this.rowBuf = ctx.createImageData(w, 1);
  this.rowData = this.rowBuf.data;
}
const rowData = this.rowData!;

    Additional note: resize() already sets this.rowBuf = null — reuse pattern was intended but not implemented in pushRow.

    C3: Ref Level Math Applied at Wrong Stage¶

    File: frontend/src/lib/renderers/waterfall-renderer.ts
    Line: 145-147

    const norm = Math.min(1.0, p / 80);
const v = Math.min(255, Math.max(0, Math.floor(Math.sqrt(norm) * 255) + refShift));

    Problem: refShift is added after sqrt curve, making it a simple brightness offset instead of proper dB gain adjustment.

    Impact: Ref level doesn't behave like a real RF gain control — affects color mapping, not signal visibility.

    Expected behavior: Adjust input p before normalization: 

    const adjusted = Math.min(80, Math.max(0, p + (refShift / 255) * 40)); // map ±127 → ±20 on 0-80 scale
const norm = adjusted / 80;
const v = Math.floor(Math.sqrt(norm) * 255);

    Severity: Medium-high — feature works but incorrectly. Users expect "Ref Level" to act like hardware gain knob.

    🟡 High Priority Issues¶

    H1: Shared Timer Variable for Multiple Long-Press Controls¶

    File: frontend/src/components/controls/FeatureToggles.svelte
    Line: 21

    let pressTimer: ReturnType<typeof setTimeout>;

    Problem: Single pressTimer shared between ATT and PRE buttons. If user: 1. Press ATT (starts timer) 2. Hold ATT 3. Press PRE (clears ATT timer, starts PRE timer) 4. ATT long-press never fires

    Impact: Edge case, but confusing UX.

    Fix: 

    let attPressTimer: ReturnType<typeof setTimeout>;
let prePressTimer: ReturnType<typeof setTimeout>;

function startAttLongPress() {
  attPressTimer = setTimeout(() => (showAttMenu = true), 500);
}
function cancelAttLongPress() {
  clearTimeout(attPressTimer);
}
// Same for PRE

    H2: Timer Cleanup Missing in Long-Press Components¶

    Files: - frontend/src/components/controls/StepSelector.svelte - frontend/src/components/controls/FeatureToggles.svelte

    Problem: If component unmounts during long-press (e.g., user navigates away), timer keeps running.

    Impact: Rare but can cause leaked refs and console warnings.

    Fix: 

    import { onDestroy } from 'svelte';

onDestroy(() => {
  clearTimeout(pressTimer);
});

    M4: Peak Hold Y-Scaling Uses Magic Numbers¶

    File: frontend/src/lib/renderers/spectrum-renderer.ts
    Line: 241-242

    const amp = Math.min(1.0, peaks[i] / 80);
const y = height * (1 - Math.sqrt(amp));

    Problem: Duplicates logic from renderSpectrum but hardcodes constants (80 max, sqrt curve). Should extract shared scaling function.

    Impact: If spectrum scaling changes, peak hold won't match.

    Fix: 

    function amplitudeToY(value: number, height: number): number {
  const amp = Math.min(1.0, value / 80);
  return height * (1 - Math.sqrt(amp));
}

    Use in both renderSpectrum and _drawPeakHold.

    💡 Lessons Learned¶

    1. .svelte.ts Files Support Runes (Svelte 5)¶

    Mistake: Review incorrectly flagged $state(0) in smoothing.svelte.ts as invalid.

    Reality: - .svelte.ts = Svelte 5 reactive module (runes allowed) - .ts = plain TypeScript (runes not allowed) - .svelte files = runes in <script> only

    Reference: Svelte 5 docs on .svelte.ts modules

    Impact: Our "fix" broke S-meter reactivity → frozen needle.

    Takeaway: Verify framework conventions before "fixing" what looks wrong. .svelte.ts is a first-class pattern in Svelte 5.

    2. Regression Testing Critical for Known Issues¶

    Problem: C2 (waterfall memory allocation) was fixed on 2026-03-08, broke on 2026-03-09.

    Why: Agent rewrote waterfall-renderer.ts without awareness of prior fix.

    Solution: - Add regression test suite for critical bugs - Include fix commit hash in code comments: // Fixed 2026-03-08 (8d5488f): reuse rowBuf - Pre-commit hook: check for known anti-patterns

    3. Multi-Agent Coordination Needs Better Protocol¶

    Observed: - Waterfall agent committed both waterfall + labels work (2 issues) - SpectrumCanvas.svelte modified by 2 agents (labels + spectrum)

    Risk: Merge conflicts, duplicate work, regression introduction

    Mitigation: - Task files should declare file ownership: FILES: waterfall-renderer.ts (exclusive) - Final reconciliation step before commit: git diff --name-only | check_conflicts
