## Architecture Review

### ✅ Strengths

1. **Minimal surface area:** Only adds HTTP/WS bridge, no protocol changes
2. **Consistent patterns:** Follows established command flow exactly
3. **Type safety:** All dataclasses, union types, pattern matching
4. **No duplication:** Reuses existing `radio.set_compressor()` from radio.py
5. **Clear separation:** Backend (poller + handlers) vs Frontend (FeatureToggles)

### ✅ Correct Design Decisions

1. **No receiver parameter:**
   - Compressor is TX-global (IC-7610 hardware)
   - Sending `receiver` would be misleading
   - Handlers like `set_power`, `ptt_on` also omit receiver

2. **Default `on=True` in handler:**
   - Matches `SetDualWatch` behavior
   - Allows simple `{"name": "set_comp"}` call (no params)
   - Toggle pattern still works: frontend always sends explicit `!comp`

3. **Placement in TX control group:**
   - Frontend puts COMP in `<ControlGroup title="TX">`
   - Makes sense: compressor only active during transmit

## Comparison to Similar Features

### SetDualWatch (similar boolean toggle)

**Similarities:**
- Both: `@dataclass` with `on: bool`
- Both: Single-param handler `params.get("on", default)`
- Both: Global (no receiver param)

**Differences:**
- DualWatch default: `False` → COMP default: `True`
- Justification: DualWatch off by default (user opt-in), COMP state query likely (toggle pattern)

**Verdict:** ✅ Acceptable difference, both reasonable defaults

---

### NB/NR/DIGISEL (per-receiver toggles)

**Difference:**
- NB/NR/DIGI: send `receiver: receiverIdx`
- COMP: no receiver param

**Why correct:**
- NB/NR/DIGI are RX path (per-receiver on IC-7610)
- COMP is TX path (global)
- IC-7610 has 2 RX but 1 TX

**Verdict:** ✅ Architecturally correct
